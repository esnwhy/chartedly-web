# ADR-008: Astro i18n with URL Prefix Routing

## Status
**Accepted** — 2026-03-24

## Context
Chartedly serves English-speaking foreigners in Japan and Japanese users. Phase 06 prepared 180+ i18n strings in JSON. The current implementation uses client-side language switching via `localStorage` + `data-i18n` attributes. Phase 07 requires proper URL-based routing (`/en/`, `/ja/`) for SEO.

## Decision
Migrate from client-side-only i18n to Astro's static i18n routing with `/en/` and `/ja/` URL prefixes. Keep the existing translations object as the string source.

## Alternatives Considered

### Client-side only (current implementation)
- **Pros:** Already built. Zero page reload. Simple toggle.
- **Cons:** Google indexes only one language (whichever renders first). No `hreflang` tags possible. URL does not reflect language — users cannot share a Japanese link. SEO is fundamentally broken for bilingual content.
- **Verdict:** Insufficient for production. Must be upgraded.

### astro-i18next
- **Pros:** Familiar i18next API, interpolation, pluralization.
- **Cons:** Additional dependency. Existing translations are in a custom TypeScript format, not i18next JSON. Would require migration of 180+ strings. Overkill — Chartedly does not use interpolation or pluralization in most strings.
- **Verdict:** Rejected. Unnecessary abstraction layer.

### Paraglide.js (inlang)
- **Pros:** Compiled translations (tree-shakeable), type-safe, small runtime.
- **Cons:** Requires `.inlang` project setup and message format migration. Newer tool with less Astro community adoption. Build step complexity.
- **Verdict:** Interesting but premature. Reconsider if i18n complexity grows.

### Astro built-in i18n routing (chosen)
- **Pros:**
  - Native Astro feature. Zero dependencies.
  - Static generation of `/en/` and `/ja/` pages at build time.
  - Proper `hreflang` tags for SEO.
  - URL reflects language — shareable, bookmarkable.
  - Existing `translations.ts` works as-is — just import and use `t('key', lang)`.
  - Content Collections can hold bilingual fields (see content-schema.md).

## Implementation

### 1. Astro Config

```javascript
// astro.config.mjs
export default defineConfig({
  i18n: {
    defaultLocale: 'en',
    locales: ['en', 'ja'],
    routing: {
      prefixDefaultLocale: true,  // /en/product/... not /product/...
      redirectToDefaultLocale: false,
    },
  },
});
```

### 2. Page Structure

```
src/pages/
  en/
    index.astro
    product/[slug].astro
    category/[slug].astro
    ranking.astro
    new.astro
    about.astro
  ja/
    index.astro
    product/[slug].astro
    category/[slug].astro
    ranking.astro
    new.astro
    about.astro
  index.astro          # redirects to /en/ (or /ja/ based on Accept-Language)
```

To avoid duplicating page logic, use a shared template:

```
src/pages/
  index.astro                    # redirect to /en/
  [...lang]/index.astro          # shared home template
  [...lang]/product/[slug].astro # shared product template
```

Or use Astro's `getStaticPaths` with locale parameter:

```astro
---
// src/pages/[lang]/product/[slug].astro
export async function getStaticPaths() {
  const products = await getCollection('products');
  const locales = ['en', 'ja'];

  return locales.flatMap(lang =>
    products.map(product => ({
      params: { lang, slug: product.id },
      props: { product, lang },
    }))
  );
}

const { product, lang } = Astro.props;
---
```

### 3. Translation Helper (updated)

```typescript
// src/i18n/utils.ts
import { translations, type Lang, type TranslationKey } from './translations';

export function t(key: TranslationKey, lang: Lang): string {
  return translations[lang][key] ?? translations['en'][key] ?? key;
}

export function getLocalizedUrl(path: string, lang: Lang): string {
  return `/${lang}${path}`;
}

export function getAlternateUrl(currentUrl: string, targetLang: Lang): string {
  return currentUrl.replace(/^\/(en|ja)/, `/${targetLang}`);
}
```

### 4. Hreflang Tags

```astro
<!-- BaseLayout.astro -->
<head>
  <link rel="alternate" hreflang="en" href={`https://chartedly.com/en${currentPath}`} />
  <link rel="alternate" hreflang="ja" href={`https://chartedly.com/ja${currentPath}`} />
  <link rel="alternate" hreflang="x-default" href={`https://chartedly.com/en${currentPath}`} />
</head>
```

### 5. Bilingual Product Content

Product JSON files gain bilingual fields (see content-schema.md):

```json
{
  "name_en": "ANESSA Perfect UV Milk",
  "name_ja": "アネッサ パーフェクトUV マイルドミルク",
  "description_en": "...",
  "description_ja": "...",
  "pros_en": ["...", "..."],
  "pros_ja": ["...", "..."]
}
```

Templates select the correct field based on `lang` parameter:

```astro
<h1>{lang === 'ja' ? product.data.name_ja : product.data.name_en}</h1>
```

### 6. Language Switcher

The existing `LanguageToggle` component changes from `localStorage` toggle to navigation:

```astro
<a href={getAlternateUrl(Astro.url.pathname, targetLang)}
   class="text-text-secondary hover:text-accent">
  {lang === 'en' ? '日本語' : 'English'}
</a>
```

No JavaScript required for language switching — it is a standard link.

## Consequences
- Every page exists at two URLs: `/en/path` and `/ja/path`. Total page count doubles.
- Build time roughly doubles (still under 20s for 116 products x 2 languages).
- SEO gets proper `hreflang` signals. Google can index both languages independently.
- The client-side `data-i18n` system in `client.ts` can be removed or kept as a progressive enhancement for strings that change dynamically (e.g., search results count).
- Product content requires bilingual fields, which means the AI enrichment pipeline must generate both EN and JA content (see content-schema.md and automation-pipeline.md).
